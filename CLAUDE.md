## Architecture

Async Python bot (asyncio + aiohttp + websockets) that trades Polymarket's 5-minute BTC Up/Down binary markets.

**Data flow**: `BinanceFeed` (WebSocket 600-tick buffer + REST) â†’ `compute_trend_context()` + `compute_signal_early()` â†’ `edge.compute_edge()` â†’ `executor` â†’ `db.py` (SQLite) + `logger.py` (thin wrapper)

### Data Storage (SQLite)

All data is stored in `bot_data.db` (SQLite). Every window observed is recorded, whether traded or not â€” this builds a dataset for backtesting and ML training.

**Tables:**
- **windows** â€” Every 5-min window observed. Resolution data (btc_open/close, winner) filled when kline fetched. Tracks momentum continuation (did winner match prior window?).
- **market_snapshots** â€” Polymarket price snapshots captured each TA iteration. Includes LMSR velocity/acceleration, Binance data, order book. Tracks up+down sum for arb detection.
- **trades** â€” All trade decisions (including SKIPs). Signal data, LMSR data at entry, execution details, outcome.
- **session_log** â€” Session metadata and final stats.
- **smc_decisions** â€” SMC strategy decision log: direction, confidence, 3-composite scores, vetoes, LMSR velocity, engine state snapshots.
- **smc_candles** â€” 1m and 5m candles generated by the SMC CandleAggregator during each window.
- **backtest_runs** â€” SMC backtester results: parameters, trade count, win rate, P&L.

CSV export available via `db.export_trades_csv()` â€” runs automatically after resolution.

### Signal Pipeline (Early Strategy)

The signal combines four components with configurable weights:

1. **Trend (40%)**: EMA(60) vs EMA(180) crossover from full tick buffer + prior 5-min window **mean reversion** (flipped â€” data shows 66% reversal rate). Don't fight the EMA trend, but fade the prior window.
2. **Micro (35%)**: RSI(7), EMA(3/7) crossover, 5s/10s momentum, volume spike. Short-term within-window signals.
3. **Order book (15%)**: Binance bid/ask volume imbalance from REST depth API.
4. **Volatility (10%)**: Stddev of 1s returns. High vol â†’ reduce signal magnitude (less confident).

The score maps to `model_prob` via confidence scaling, then `edge.py` compares against Polymarket token prices.

### LMSR Price Velocity Strategy (`--strategy lmsr`)

Uses ONLY Polymarket price dynamics â€” no Binance TA. Detects informed money flow by tracking how fast token prices move.

1. At T+5s: Start collecting market snapshots every 2s
2. Track `up_price` velocity ($/sec) and acceleration ($/secÂ˛)
3. At T+15s to T+25s: If velocity > threshold AND price < $0.60, follow the smart money
4. Skip if velocity is low (nobody trading) or prices already one-sided (> $0.60)

**Key insight**: When velocity is high and acceleration is positive (move is speeding up), informed money is flowing in. This aggregates all other participants' information.

### Selective Strategy (`--strategy selective`)

Data-driven strategy combining two empirical signals that must agree (confluence):

1. **Mean reversion** (66% reversal rate): If prior window was UP, expect DOWN and vice versa. Requires prior move > 0.03% to qualify.
2. **LMSR velocity** (57% directional accuracy at high velocity): Follow the direction that smart money is pushing prices.
3. **Only trade when BOTH agree**: Prior window reversal direction must match LMSR velocity direction. Skip otherwise.

**Skip conditions** (designed to skip ~50-60% of windows):
- No prior window data (first window after launch)
- Prior window move too small (< `SELECTIVE_MIN_PRIOR_MOVE`)
- LMSR velocity too low (< `LMSR_VELOCITY_THRESHOLD`)
- No confluence (reversal and LMSR disagree)
- Price already too expensive (> `LMSR_MAX_PRICE`)

This is the highest-conviction strategy â€” fewer trades, higher expected win rate.

### SMC Strategy (`--strategy smc`)

Smart Money Concepts (SMC) structural analysis using a 3-composite scoring model + LMSR velocity confluence. This is the most sophisticated strategy, using price structure rather than traditional TA indicators.

**Architecture â€” 3-Composite Scoring Model:**
1. **Momentum (40%)**: LMSR velocity signal, BOS type (impulsive vs corrective), order flow count, multi-timeframe trend alignment
2. **Structure (35%)**: Control state (supply/demand in control), zone position (premium/discount), swing strength, return type, zone quality
3. **Confluence (25%)**: Liquidity sweeps, supply-demand flips, Quasimodo patterns, FVG fills, engulfing candles

**Timing within a 5-minute window:**
- T+0sâ€“T+60s: Feed 1s ticks to CandleAggregator, build first 1m candle
- T+5s: Start collecting LMSR market snapshots every 2s
- T+60sâ€“T+180s: 1m candles complete, SMC engines build market structure (swings, BOS, zones)
- T+180sâ€“T+250s: Decision window â€” DecisionEngine evaluates every 2s
- T+250s: Hard deadline

**Engine pipeline per 1m candle:**
`CandleAggregator` â†’ `MarketStructure` (swings, BOS, trend) â†’ `ZoneEngine` (supply/demand zones) â†’ `ControlState` (who's in control) â†’ `LiquidityEngine` (EQH/EQL, sweeps) â†’ `ConfluenceEngine` (FVG, QM, flips, engulfing)

At decision time: `ScoringEngine` scores both directions â†’ `DecisionEngine` picks direction, applies vetoes, sizes bet â†’ `SMCTradeLogger` records everything.

**SMC Engine Files (`engines/`):**
- **candle_aggregator.py** â€” Converts 1s ticks to 1m and 5m candles via `feed_tick(price, timestamp)`
- **market_structure.py** â€” Swing detection (HH/HL/LL/LH), BOS/CHoCH detection, trend classification
- **zone_engine.py** â€” Supply/demand zone creation from BOS events, freshness tracking, quality scoring
- **control_state.py** â€” State machine: supply_in_control / demand_in_control / neutral
- **liquidity_engine.py** â€” Equal highs/lows (EQH/EQL) detection, sweep detection
- **confluence_engine.py** â€” FVG detection/fill, Quasimodo patterns, S/D flips, engulfing candles, return type classification
- **scoring_engine.py** â€” 3-composite model scoring with configurable weights
- **decision_engine.py** â€” Direction selection, veto system, confidence/sizing, `BetDecision` output
- **smc_trade_logger.py** â€” Logs decisions and candles to `smc_decisions` and `smc_candles` DB tables
- **backtester.py** â€” Replays historical data through SMC pipeline, stores results in `backtest_runs` table

**Supporting Files:**
- **utils/candle_types.py** â€” All SMC data types: `Candle`, `Direction`, `SwingPoint`, `BOS`, `Zone`, `BetDecision`, `ScoreBreakdown`, enums
- **smc_config.py** â€” All tunable SMC parameters (swing lookback, BOS thresholds, zone freshness, confidence thresholds, scoring weights)

**SMC Database Tables:**
- **smc_decisions** â€” Every decision evaluation: direction, confidence, scores (momentum/structure/confluence), vetoes, LMSR velocity, engine state
- **smc_candles** â€” All 1m and 5m candles generated during SMC windows
- **backtest_runs** â€” Backtester results: parameters used, trade count, win rate, P&L

### Five Strategy Modes (`--strategy early|late|lmsr|selective|smc`)

**Early entry (default)**: Trades at T+10s to T+30s. Composite TA signal with mean reversion from prior window. Trades ~70% of windows.

**Late entry**: Original strategy at T-10s to T-5s. RSI(14), EMA(9/21). Problem: markets are $0.99/$0.01 by then.

**LMSR**: Pure market microstructure at T+15s to T+25s. Follows Polymarket price velocity. No warmup needed.

**Selective**: Reversal + LMSR confluence at T+15s to T+25s. Highest conviction â€” trades only when prior window reversal AND LMSR velocity agree. Targets 55-65% win rate on ~40-50% of windows.

**SMC**: Full structural analysis at T+180s to T+250s. Builds 1m candles from ticks, runs through complete SMC pipeline (market structure, zones, liquidity, confluence), scores with 3-composite model, uses LMSR velocity as momentum input. Most data-rich strategy â€” needs 3+ minutes of candle data. No warmup needed.

### Outcome Resolution

ALL observed windows are resolved (not just traded ones) to build a complete dataset. `resolve_pending_windows()` fetches Binance 5m klines for closed windows, updates both the windows and trades tables. Momentum continuation is computed by comparing each window's winner to the prior window's winner.

### Session End Analysis

At session end, four analyses are printed:
1. **Session summary** â€” Trades, win rate, P&L, windows observed/resolved
2. **Momentum continuation** â€” What % of windows continue prior direction (validates reversal signal)
3. **LMSR velocity analysis** â€” When velocity is high, does direction match winner?
4. **Velocity threshold analysis** â€” Accuracy at different velocity thresholds (0.005â€“0.030) to calibrate the threshold

## Key Domain Details

- **Window timing**: `ts = now - (now % 300)`, slug = `btc-updown-5m-{ts}`
- **Tie rule**: end >= start â†’ Up wins (+0.005 bias in `model_prob`)
- **Gamma API `clobTokenIds`**: may be JSON string â€” parsed with `json.loads` fallback
- **CLOB `get_price()`**: returns `{"price": "0.52"}` dict; `{"price": "0"}` when empty â†’ midpoint fallback
- **Order constraints**: min 5 shares, maker amount 2 decimal places
- **LMSR up+down sum**: If < 0.98, potential arb opportunity â€” logged in snapshots for analysis
- **Momentum reversal**: Unreliable across samples â€” first 30-window sample showed 66% reversal, second showed 52% (essentially random). Not statistically significant as a standalone signal.

## Key Design Decisions

- **Mean reversion is unreliable**: Two 30-window samples showed 66% and 52% reversal rates â€” the difference is not statistically significant and the signal is essentially noise. Do not rely on reversal as a primary signal. LMSR velocity is the strongest validated signal.
- **Trend EMA > Micro**: At 5-minute binary markets, EMA(60/180) crossover is the strongest within-window predictor. Micro TA (RSI, short EMAs) is noise without trend context. Weight allocation: 40% trend, 35% micro.
- **Cold start**: First window after launch may lack trend data (need 60+ ticks). Confidence will be lower, bot may skip. This is correct â€” better to skip than trade blind. LMSR/selective/SMC strategies have no warmup requirement.
- **Selective confluence underperformed**: The reversal signal's unreliability caused the confluence filter to reject good LMSR trades. Pure LMSR with a calibrated threshold is the preferred approach. Selective strategy is retained but not recommended.
- **Confidence filter**: Prevents trading when signals disagree or are weak. Reduces trade frequency by ~30% but improves win rate on remaining trades.
- **Price range gate**: Early markets start near 50/50 but move fast. By T+15s markets can reach $0.70/$0.30. Gate ensures we only enter at fair prices.
- **Resolution via Binance kline**: Uses Binance 5m candle open/close as proxy for Chainlink oracle. Accurate for backtesting; rare divergence possible.
- **Record everything**: Every window is observed and recorded, even if not traded. Market snapshots captured every TA iteration. This builds a complete dataset for backtesting, ML training, and strategy calibration.
- **LMSR velocity threshold**: Backtesting on 30 windows shows 0.012 as the sweet spot (57% win rate, best P&L/trade). Aggregate velocity analysis overstates accuracy vs actual simulated entry. Session-end threshold analysis is directional but not a substitute for entry-level backtesting.
- **Database is source of truth**: SQLite `bot_data.db` stores all data. CSV export is for convenience only.

## Common Issues

- **`round(0.005, 2)` â†’ 0.0**: `risk.py` uses `math.ceil(amount * 100) / 100`.
- **Sync CLOB in async loop**: `get_market_prices()` wrapped in `run_in_executor()`.
- **First window skipped**: Normal on cold start â€” not enough ticks for trend context. Does not apply to LMSR/selective/SMC strategies.
- **SMC skips early in window**: Expected â€” SMC needs 3+ completed 1m candles before the decision engine activates at T+180s. If fewer than 3 candles complete, the window is skipped.
- **Selective skips most windows**: Expected behavior. Skip rate of 50-60% means the strategy is being selective. Check skip reasons in logs â€” "no confluence" is the most common and correct.
- **All iterations filtered**: Either prices moved outside range (market priced in direction fast) or confidence too low (signals disagree). Both are correct skip behavior.
- **Database locked**: SQLite uses WAL mode for concurrent reads. Only one writer at a time â€” should not be an issue for single-process bot.
